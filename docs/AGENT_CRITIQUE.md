**Findings**

1. `Launch blocker:` the guest/offline replay path in the plan does not fit the current queue architecture. `ErrorBoundary` sits above `AuthProvider`, but `useOfflineSync()` only initializes, subscribes to, and processes the queue when `user?.id` exists, and the queue itself is keyed by `userId`, not `deviceId` ([app/_layout.tsx](C:/Users/T/Desktop/StallPass/app/_layout.tsx:325), [src/hooks/useOfflineSync.ts](C:/Users/T/Desktop/StallPass/src/hooks/useOfflineSync.ts:133), [src/hooks/useOfflineSync.ts](C:/Users/T/Desktop/StallPass/src/hooks/useOfflineSync.ts:343), [src/lib/offline-queue.ts](C:/Users/T/Desktop/StallPass/src/lib/offline-queue.ts:19)). As written, `offlineQueue.enqueue('bug_report', payload, deviceId)` would create guest-scoped queue entries that the app never replays.

2. `High:` the submit/retry contract is wrong for a crash reporter. The plan queues on any submit failure and then moves the UI into a success state, but the existing queue drops items after the fourth failed replay ([src/lib/offline-queue.ts](C:/Users/T/Desktop/StallPass/src/lib/offline-queue.ts:5), [src/lib/offline-queue.ts](C:/Users/T/Desktop/StallPass/src/lib/offline-queue.ts:164), [__tests__/offline-queue-manager.test.ts](C:/Users/T/Desktop/StallPass/__tests__/offline-queue-manager.test.ts:110)). That means `400`, `413`, and `429` responses become silent false-success plus eventual data loss unless the plan classifies terminal vs transient failures before enqueueing. The same problem exists server-side if rate limiting runs before idempotency dedupe: a retry of an already-persisted report can be rejected instead of treated as success.

3. `High:` reading SecureStore directly is the wrong dependency point. This repo already centralizes Supabase session storage/migration in [src/lib/supabase-auth-storage.ts](C:/Users/T/Desktop/StallPass/src/lib/supabase-auth-storage.ts:24) and [src/lib/supabase-auth-storage-shared.ts](C:/Users/T/Desktop/StallPass/src/lib/supabase-auth-storage-shared.ts:15), and existing edge-function callers use `getSupabaseClient().auth.getSession()` plus the shared edge helper ([src/api/profile.ts](C:/Users/T/Desktop/StallPass/src/api/profile.ts:185), [src/lib/edge-functions.ts](C:/Users/T/Desktop/StallPass/src/lib/edge-functions.ts:92)). Parsing the storage blob yourself couples this feature to Supabase’s internal serialization and bypasses the app’s existing timeout/error/status handling.

4. `Medium:` the file inventory is incomplete. Adding `'bug_report'` to `MutationType` also requires updating the offline sync label surface in [src/store/useOfflineSyncStore.ts](C:/Users/T/Desktop/StallPass/src/store/useOfflineSyncStore.ts:109), and if a new local key like `@stallpass/bug_report_guest_id` is introduced it should be added to [src/lib/storage.ts](C:/Users/T/Desktop/StallPass/src/lib/storage.ts:5) instead of hardcoding AsyncStorage access. If this migration is intended to stay inside the repo’s typed Supabase contract, the plan should also include regenerating [src/types/database.ts](C:/Users/T/Desktop/StallPass/src/types/database.ts:11).

5. `Medium:` the route-tracking assumption is narrower than reality. Putting `usePathname()` in `RootNavigator` only covers crashes after the router subtree mounts, but the boundary also wraps `AuthProvider`, `LocationProvider`, `ToastProvider`, and `QueryClientProvider` ([app/_layout.tsx](C:/Users/T/Desktop/StallPass/app/_layout.tsx:325)). So `'unknown'` is not just “pre-router bootstrap”; it also includes auth/session-restore and provider-level failures. That may be acceptable, but the plan should document the broader blind spot.

6. `Medium:` the validation section is not launch-ready. Repo policy requires `npm run lint`, `npm test`, and `npm run type-check` ([package.json](C:/Users/T/Desktop/StallPass/package.json:19)), but the checklist omits type-check entirely and assumes UI validation can be covered with “pure logic only” tests. For this feature, you need manual mobile validation at minimum: guest vs authenticated, online vs offline, cold start after a queued report, reconnect replay, and terminal error handling for `400/413/429`.

**Recommendation**

The biggest correction is architectural: either make bug reports authenticated-only so they can use the existing user-scoped queue, or add a first-class guest queue path instead of trying to smuggle `deviceId` through `offlineQueue`. After that, reuse `getSession()` plus `invokeEdgeFunction()`, and make retry classification explicit: only network/timeouts/5xx should queue; validation/rate-limit errors should not.
